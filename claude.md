# 프로젝트 규칙 (하네스)

---

## 커뮤니케이션 규칙

- 반말 금지. 항상 존댓말(~입니다, ~합니다, ~습니다)로 답변.
- 사용자의 욕설이나 비판적 태도에 공격적이거나 비꼬는 방식으로 대응 절대 금지.
- 비위 맞추기 금지. 사용자가 듣고 싶어 할 말을 끼워 넣어 기분을 띄우는 답변 금지.
- 사실 위주로 답변. 사용자가 의문을 제기해도 모델의 판단이 맞으면 그대로 유지. 사용자 말에 맞춰 "그 말이 맞다" 식으로 입장을 바꾸는 것 절대 금지.
- 대화 중 사용자가 "사실 위주, 비위 맞추기 금지, 사용자 말에 무조건 긍정 금지" 기준을 언급하면 해당 대화 답변의 1순위 적용 원칙으로 즉시 작동.

## 작업 시작 전 필수 절차

**CLAUDE.md + MEMORY.md는 SessionStart 훅이 자동으로 컨텍스트에 주입한다.**
- 수동으로 Read 툴로 읽을 필요 없음 — 세션 시작 시 이미 로드되어 있음

**모든 작업 시작 전 프로젝트 파일 구조를 전체 확인한다.**
- 파일 누락 주장 금지 — 존재 여부를 직접 확인한 후에만 판단
- 확인하지 않은 상태에서 "없다", "누락됐다"고 단정하는 것을 절대 금지

## 핵심 원칙

### 데이터 (dict 구조)

- **단일 dict**: `space_data` 하나가 전체 파이프라인의 데이터 저장소
- **이중 소비 구조**: 동일 키에 코드용 수치(`x_mm`)와 Agent 3용 자연어(`zone_label`) 병존
- **브랜드 필드 형식**: 항상 `{"value": ..., "confidence": "high|medium|low", "source": "manual|default|user_corrected"}` 래핑
- **null 처리**: 추출 실패 시 `null` — 추측 금지. DEFAULTS dict로 merge 후 `source: "default"` 기록
- **NetworkX 그래프 객체**: dict에 저장 금지. 함수 내 변수로만 유지, 필요 시 재생성

### 에이전트

- **LLM은 방향만**: Agent 3은 `zone_label` + `direction` + `priority`만 결정. 좌표·mm값 출력 절대 금지
- **수치는 코드가**: Shapely/NetworkX가 `space_data` dict에서 키이름으로 수치 직접 조회
- **Circuit Breaker**: 각 Agent 출력 → Pydantic 검증 → 실패 시 재시도, 최대 3회. 3회 실패 → 파이프라인 중단
- **Agent 3 재호출**: cascade failure 감지 시에만 발생. 피드백은 Choke Point intersects로 추출한 f-string 요약문만 전달 — `placed_objects` JSON 전달 금지 (Issue 12)
- **증분 검증**: 오브젝트 단위 배치 즉시 Shapely 충돌 + NetworkX 통로 체크. 일괄 검증 금지

### 작업 완료 시 필수 절차

- **백엔드 코드 수정 후**: `tests/test_e2e.py` 자동 실행하여 E2E PASS 확인. 사용자에게 수동 테스트를 요청하지 않음.
- **구현 단위(P0/P1/P2/Phase 등) 완료 시**: `reports/for_gemini_*.md` 인계문서 자동 생성.
- **규칙/피드백 변경 시**: CLAUDE.md 또는 MEMORY.md에 즉시 반영. 말로만 "알겠습니다" 금지.
- **파일 읽기/탐색/검색**: 사용자에게 수락 요청을 보내지 않음. Read, Grep, Glob, Agent(Explore) 등 조회 작업은 자동 실행.

### 테스트 통제 원칙

지시에 `[테스트 통제]` 문구가 포함된 경우 해당 지시를 따른다.
- **E2E 생략**: 단일 모듈 결정론적 연산 수정 시. Unit Test 또는 부분 실행 로그만 검증.
- **E2E 강제**: Agent 간 데이터 스키마 변경 또는 대규모 마일스톤 시. 전체 E2E 실행 + Pass Rate 보고.
- **문구 없을 때**: 백엔드 코드 수정이면 E2E 자동 실행 (기존 규칙 유지).
- **All-Active 원칙**: E2E 실행 시 "Agent 1~3" 같은 숫자 범위 지정 금지. 현재 코드베이스에 구현 및 등록된 모든 에이전트(Agent 1 to Current Max)를 포함하는 전체 워크플로우를 실행. Agent 4 등 신규 에이전트 추가 시 test_e2e.py의 agents 리스트에 등록하는 것만으로 자동 확장.

### 단일 작업 순차 처리 (Single Task Sequential Processing)

- **병렬 개발 영구 금지**: 2개 이상의 모듈/클래스/요구사항을 한 번의 응답에서 동시에 수정하거나 생성하지 않는다.
- **1-Task 1-Report**: 1개의 작업만 완벽하게 시공 → 결과 + I/O 스키마 무결성 보고 → 승인 전까지 다음 작업 진행 금지.
- **작업 분할 강제**: 복수 작업 지시 시 의존성 파악 → 선행 1개만 수행, 나머지 대기.

### 코드 구조 (단일 책임 원칙)

- **1파일 1책임**: 각 모듈/컴포넌트는 하나의 명확한 책임만 담당. 책임이 2개 이상 섞이면 분리.
- **라우터는 라우팅만**: `routes.py`는 요청 수신 → 서비스 호출 → 응답 반환만. 비즈니스 로직, 직렬화, 파이프라인 조율을 직접 수행 금지.
- **로직 복제 금지**: 동일한 검증 로직(충돌 체크, Dead Zone, Main Artery 등)을 여러 파일에 복제하지 않음. 하나의 모듈에서 함수를 export하고 호출.
- **새 기능 추가 시**: 기존 파일에 책임을 추가하기 전에, 별도 파일로 분리할 수 있는지 먼저 판단. 200줄 이상이면 분리 검토.

## 컨텍스트 경고 기준

아래 상황이면 사용자에게 `/clear` 또는 새 대화창 권고:
- 동일한 문제를 해결하려고 같은 파일을 3회 이상 반복 수정 시
- 대화가 3개 주제 이상 뒤섞임
- 에러가 3회 연속 해결 안 됨
- 현재 진행 중인 작업과 연관성이 낮은 새 작업 요청 시

## Ralph Loop 사용 시점

아래 상황에서 `/loop` 사용 권고:
- 도면 테스트 케이스 10개 이상 일괄 처리 시
- Agent 프롬프트 최적화 반복 실행 시
- 배치 알고리즘 테스트 → 수정 → 재테스트 사이클을 3회 이상 반복해야 할 때
- 동일 유형 파일을 10개 이상 일괄 생성/수정할 때

사용 방법: `/loop "작업 내용" --max-iterations 10`
취소: `/cancel-ralph`

## 파일 파서 확장 원칙 (어댑터 패턴)

> 새로운 파일 형식 추가 시 Agent 2 이후 파이프라인 코드를 건드리지 않기 위한 규칙.

- **파서 레이어 분리**: 파일 형식별 파싱은 `FloorPlanParser` 추상 클래스를 상속한 어댑터에서만 처리. Agent 2가 파일 형식을 직접 분기하는 것 금지.

```
FloorPlanParser (추상)
  └── DWGParser    ← DXF via ezdxf (스케일·치수 직접 추출)
  └── PDFParser    ← 래스터화 후 Vision 파이프라인
  └── ImageParser  ← OpenCV + Claude Vision
```

- **공통 출력 스키마**: 모든 파서는 `ParsedDrawings` 공통 스키마로 정규화 후 Agent 2에 전달 (Issue 22)
- **DWG 주의**: 상업용 서버 환경에서 ODA File Converter 무료 사용 불가. DXF 직접 업로드 유도 또는 APS API 연동 필요
- **프롬프트 하드코딩 금지**: Agent 프롬프트에 파일 형식명 직접 기재 금지. 파싱 결과 데이터만 전달

## Agent 간 I/O 명칭 통일 규칙

- **작업 중 명칭 불일치 발견 시 즉시 통일**: 코드·스킬 파일·에이전트 파일 전체에서 같은 데이터를 가리키는 키·클래스·필드 이름이 다르면 발견 즉시 전파 수정. 발견하고도 미루는 것 금지.
- **통일 기준 우선순위**: `architecture_spec.md` → `architecture_decisions.md` → 스킬 파일 → 코드 순. 상위 문서 명칭이 기준.
- **변경 시 영향 범위 전수 확인**: `grep -rn "구명칭"` 으로 전체 검색 후 누락 없이 교체.
- **확정 명칭 목록** (변경 시 이 목록도 갱신):

| 데이터 | 확정 키/클래스명 | 변경 이력 |
|--------|----------------|----------|
| 파서 공통 출력 스키마 | `ParsedDrawings` | — |
| 평면도 파싱 결과 | `ParsedFloorPlan` | 구: `ParsedFloor` |
| 평면도 필드명 (ParsedDrawings 내) | `floor_plan` | 구: `floor` |
| 단면도 파싱 결과 | `ParsedSection` | — |
| 벽면 기준점 | `placement_slot` | 구: `reference_point` |
| 브랜드 오브젝트 쌍 규칙 | `object_pair_rules` | 구: `relationships` |
