## 1. 서비스 개요

팝업스토어가 설치될 **도면(이미지·PDF)** 과 **브랜드 메뉴얼(PDF)** 을 입력하면 AI Agent가 공간 구조를 분석하고, 시공물·조형물을 Whitebox 형태의 3D 초안으로 자동 배치하는 서비스.

- **포지션**: "빈 도면에서 설계사 검토 가능한 Whitebox 초안"까지 30분 이내
- **출력물**: Whitebox 3D 초안 + SketchUp Pro 호환 .glb + 설계 근거 리포트
- **역할 범위**: 초안 자동 생성. 실시설계 및 최종 검토는 설계사가 SketchUp에서 수행

## 3. 서비스 전체 흐름

### 3-1. 파이프라인 개요

```
[입력 화면]   도면 + 메뉴얼 업로드
    ↓
Agent 1   브랜드 메뉴얼 수치 추출 + 사용자 확인              [LLM]
    ↓  [dict 저장]
Agent 2 전반부   OpenCV + OCR + Vision 자동 감지             [LLM + 코드]
    ↓  [임시 저장]
[사용자 마킹]   입구 확인 + 미감지 설비 추가
    ↓
Agent 2 후반부   마킹 합산 → Dead Zone + NetworkX 계산       [코드]
    ↓  [dict 저장 완료]
── 1단계 사용자 확인 (결과 확인만) ──
    ↓
[오브젝트 선별 모듈]   dict + DB 기반 필터                   [코드]
    ↓
Agent 3   배치 의도 결정 (자연어 판단 / Pydantic 출력 강제)  [LLM]
    ↓  [Pydantic 검증]
[Shapely 배치 계산 + 증분 NetworkX 검증]                     [코드]
    실패 → 코드 레벨 위치 조정 → 불가 시 Agent 3 재호출
    ↓
[검증 모듈]   Dead Zone + 소방/통로 최종 확인                [코드]
    ↓
[구조화 모듈]   Three.js → .glb                              [코드]
    ↓
Agent 4   레이아웃 검토 (MVP 후순위)                         [LLM]
    ↓
Agent 5   리포트 (MVP: 템플릿)                               [템플릿]
```

### 3-2. dict 구조

Agent 1과 Agent 2의 출력을 단일 dict에 저장. 소비자에 따라 두 형태로 분리.

```
코드용 dict (Shapely/NetworkX가 수치 직접 조회):
  space_data["floor"]["polygon"] = [[0,0],[6000,0],...]
  space_data["entrance"]["x_mm"] = 0
  space_data["north_wall_mid"]["x_mm"] = 2300
  space_data["sprinkler_1"]["radius_mm"] = 2300

Agent 3 판단용 dict (자연어 요약):
  space_data["entrance"]["zone_label"] = "entrance_zone"
  space_data["north_wall_mid"]["shelf_capacity"] = 3

브랜드 제약 dict:
  space_data["brand"]["clearspace_mm"] = {"value": 1500, "confidence": "high", "source": "manual"}
  space_data["brand"]["relationships"] = [{"rule": "라이언과 춘식이 떨어뜨릴 것"}]
```

### 3-3. 입력 화면 + 브랜드 확인

```
도면 업로드 + 메뉴얼 업로드
→ Agent 1 실행 → 추출 결과 즉시 표시
→ 사용자 확인/수정
→ null 항목 → 기본값 적용 (source: "default")
→ confidence low → 하이라이트 표시
→ [다음] → Agent 2 시작
```

브랜드 제약은 도면과 무관한 정보이므로 입력 시점에서 확인. 도면 마킹 단계와 관심사 분리.

### 3-4. 사용자 마킹 (Agent 2 전반부 이후)

```
도면 위 오버레이:
  자동 감지 항목 → 확인/수정 (드래그)
  미감지 항목 → 클릭으로 추가 또는 "모르겠음"

감지됨:
  → 사용자 확인 후 Agent 2 후반부로 전달

미감지 + 사용자 마킹:
  → confidence: user_input → Dead Zone 생성

미감지 + 모르겠음:
  → disclaimer 목록 → 면책 조항
```

마킹이 계산 이전이므로 Dead Zone 재계산 없이 한 번에 처리.

### 3-5. 브랜드 메뉴얼 추출 전략

```
추출 대상 (5개만):
  1. clearspace_mm (여백/이격/띄움/여유 공간) → mm 통일
  2. character_orientation (배치 방향/정면) → 정규화
  3. prohibited_material (금지 소재) → 소재명
  4. logo_clearspace_mm (로고 여백) → mm 통일
  5. relationships (관계 제약) → 자연어 그대로

프롬프트 규칙:
  - 캐릭터명/IP명/마스코트명 → character_bbox 간주
  - 무명 캐릭터도 문맥("조형물", "피규어") 기반 판별
  - 동의어 전부 처리 ("이격" = "띄움" = "여백" = "멀리")
  - 없으면 null (추측 금지)
  - confidence 반환 (high/medium/low)

오탈자 방어:
  Pydantic 값 범위 validator (clearspace 300~5000mm)
  범위 밖 → 사용자 확인 요청

실패 처리:
  null → DEFAULTS merge (clearspace 1000mm 등)
  source 필드로 "메뉴얼 추출" vs "기본값" vs "사용자 수정" 구분
```

### 3-6. 증분 NetworkX 검증 구조

(v15.1과 동일 — 변경 없음)

```
오브젝트 1 배치 시도
    → Shapely: 충돌 없는가?
    → NetworkX: 통로 900mm 유지되는가?
        둘 다 통과 → 좌표 확정 → 다음 오브젝트
        실패 → 코드 레벨 위치 조정 시도
            조정 성공 → 좌표 확정 → 다음 오브젝트
            조정 실패 → Agent 3에 피드백 + 재호출 (최대 2회)
```

### 3-7. Agent 3 수치 차단 구조

(v15.1과 동일 — 변경 없음)

```
프롬프트: 자연어 요약 제공 + "좌표·mm값 출력 금지" 명시
출력 스키마: reference_point 키이름 + direction만 허용 (수치 필드 없음)
Pydantic: 스키마 위반 시 Circuit Breaker 차단
```

### 3-8. Agent 3 재호출 피드백 (va 추가)

```
feedback = {
    "failed_object": "photo_zone",
    "reason": "corridor_blocked",
    "placed_objects": [
        {"object_type": "character_bbox", "reference_point": "entrance"},
        {"object_type": "shelf_rental", "reference_point": "north_wall_mid"}
    ],
    "alternative_references": [...]
}

placed_objects: 이미 점유된 reference_point 목록
  → Agent 3이 점유된 곳에 재배치하는 실수 방지
  → Pydantic validator에서도 점유 키 제외 가능
```

### 3-9. 도면 입력 정책

```
설비 기호 자동 감지됨:
  → 사용자 확인 → Dead Zone 생성 → 정상 검증

설비 기호 미감지 + 사용자 마킹:
  → Dead Zone 생성 → 정상 검증

설비 기호 미감지 + "모르겠음":
  → Dead Zone 없이 진행
  → 리포트에 면책 조항 강제 삽입

사용자 직접 입력:
  → confidence: user_input → Dead Zone 생성
```

### 3-10. violation severity

```
blocking: Dead Zone 침범 / 소방 대피로 미확보 / 주 통로 900mm 미달
  → .glb 출력 차단 + 재설계 요청

그 외: .glb 정상 출력
  (MVP에서는 blocking/통과 2단계. warning/info 세분화는 이후)
```

### 3-11. JSON Circuit Breaker

(v15.1과 동일)

```
각 Agent 출력 → Pydantic 스키마 검증
    ├─ 통과 → 다음 단계
    └─ 실패 → 재시도 프롬프트로 LLM 재호출
               최대 3회 재시도
               3회 실패 → 파이프라인 중단 + 사용자 오류 알림
```

## 4. Agent 및 모듈 상세

### Agent 1 — 브랜드/기준법 수치 추출 (선행)

```
입력: 브랜드 메뉴얼 PDF (Claude Document API)

추출 대상:
  clearspace_mm / character_orientation / prohibited_material
  logo_clearspace_mm / relationships

소방법/시공기준: 하드코딩 (법 고정값)
  main_corridor_min_mm: 900
  emergency_path_min_mm: 1200
  wall_clearance_mm: 300

출력: dict 저장 (confidence + source 포함)
  null → 기본값 merge
  오탈자 → 값 범위 validator 방어
```

### Agent 2 전반부 — 자동 감지

```
입력: 도면 이미지/PDF + Agent 1 dict

처리:
  OpenCV → polygon 추출
  OCR → 스케일 계산 (scale_confidence)
  Claude Vision → 입구 + 스프링클러 + 소화전 + 분전반 동시 감지

출력: 임시 저장 (auto_detected)
  계산 안 함. 사용자 마킹 대기.
```

### [사용자 마킹]

```
자동 감지 확인/수정 + 미감지 설비 추가
마킹 완료 → Agent 2 후반부로 전달
"모르겠음" → disclaimer 목록
```

### Agent 2 후반부 — Dead Zone + 기준점 + NetworkX

```
입력: auto_detected (마킹 반영) + Agent 1 dict

처리:
  px → mm 변환
  Shapely: Dead Zone + reference_point 계산
  NetworkX: 격자 그래프 + 보행 거리

출력: dict 저장 (코드용 수치 + Agent 3용 자연어 요약)
  재계산 없음 (마킹이 계산 이전이므로)
```

### [1단계 사용자 확인]

```
결과 확인만. Dead Zone + reference_points + eligible_objects 시각화.
"진행" → Agent 3 / "수정" → 마킹 단계로 복귀
```

### [오브젝트 선별 모듈]

```
입력: dict(공간 수치) + Agent 1 dict(제약) + Supabase furniture_standards
처리: bbox 필터 + 브랜드 금지사항 교차 검증
출력: eligible_objects (bbox 포함)
```

### Agent 3 — 배치 의도 결정

```
입력: Agent 3 판단용 dict (자연어) + eligible_objects + 관계 제약
      + "좌표·mm값 출력 금지" 명시

출력 스키마:
  object_type / reference_point / direction / priority / placed_because
  수치 필드 없음 → Pydantic 강제
```

### [Shapely 배치 계산 + 증분 NetworkX 검증]

```
Agent 3 키이름으로 dict에서 수치 직접 조회
오브젝트 단위 배치 + 즉시 통로 체크
실패 → 코드 위치 조정 → 불가 시 Agent 3 재호출 (placed_objects 포함)
```

### [검증 모듈]

```
blocking → .glb 차단
그 외 → .glb 정상 출력
```

### [구조화 모듈] — Three.js

```
layout_objects → Whitebox 3D
높이: DB 저장값 사용 (2D 배치 + DB 높이 조합)
높이 간섭: SketchUp에서 확인 (리포트 기재)
.glb export
```

### Agent 4 — 레이아웃 검토 (MVP 후순위)

```
MVP에서 제외. 스프린트 5 이후 검토.
Shapely가 수치 검증을 이미 처리하므로 MVP 파이프라인 동작에 영향 없음.
```

### Agent 5 — 리포트 (MVP: 템플릿)

```
MVP: 템플릿에 값 끼워넣기
  배치 근거: Agent 3 placed_because
  검증 결과: 검증 모듈 violations
  source별 표기: "메뉴얼 추출" / "기본값" / "사용자 수정"
  면책 조항: disclaimer 자동 삽입

이후: LLM으로 자연어 매끄럽게 변환
```

## 5. 기술 스택

| 영역 | 기술 | 비고 |
|------|------|------|
| 프론트엔드 | React + TypeScript | Three.js 3D 캔버스 |

| 배포 | AWS EC2 t3.large | Docker Compose |

## 8. 미해결 논의점

### 논의점 1 — 단위 변환 상수 (스프린트 2 기획회의)

```
선택지 A: 1 unit = 1mm
선택지 B: 1 unit = 100mm
결정 후 geometry_utils.js MM_TO_UNIT 고정
```

### 논의점 2 — 배치 실패 후 부분 출력 기준

몇 개 이상 배치 성공 시 출력할 것인가 기준 미정.

### 논의점 3 — Agent 4 피드백 재계산 범위
