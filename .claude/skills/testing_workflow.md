# 테스트 및 이슈 추적 스킬 (Testing & Issue Tracking Skill)

## 목적
모든 작업 분기(기능 구현, 버그 수정, 설계 변경)에서 테스트 결과를 기록하고, 발견된 문제를 이슈로 분류·추적.

> ⚠️ 이 스킬은 자동 적용 — 별도 지시 없이도 작업 분기마다 테스트 후 기록

---

## 폴더 구조

```
reports/
├── test_results/    ← 테스트 실행 결과 기록
└── issues/          ← 발견된 문제 + 해결 기록
```

---

## 1. 테스트 결과 기록 (`reports/test_results/`)

### 실행 시점
- 작업 분기 완료 시 (기능 구현, 버그 수정, 설계 변경 후)
- 코드 변경이 있는 모든 PR/커밋 단위

### 파일명 규칙
```
{날짜}_{task_name}.md
```
예: `2026-04-02_buffer_validation.md`, `2026-04-03_agent3_circuit_breaker.md`

### 템플릿

```markdown
# 테스트 결과: {task_name}

- 날짜: {YYYY-MM-DD}
- 시간: {HH:MM}
- 횟수: {N}회차
- 대상: {테스트 대상 모듈/기능}

## 테스트 내용

{무엇을 테스트했는지}

## 결과

| 항목 | 기대 결과 | 실제 결과 | 통과 |
|------|-----------|-----------|------|
| ... | ... | ... | ✅/❌ |

## 발견된 문제

- 없음 / {문제 요약 + → issues/{파일명}.md 참조}
```

---

## 2. 이슈 관리 (`reports/issues/`)

### 이슈 생성 시점
- 테스트 결과에서 ❌ 항목 발생 시
- 구현 중 설계 충돌 발견 시
- 예상치 못한 동작 발견 시

### 파일명 규칙
```
미해결: {task_name}.md
해결됨: {task_name}_solved.md    ← 파일명 변경 (이동 아님)
```
예: `zone_split_deadlock.md` → `zone_split_deadlock_solved.md`

### 미해결 이슈 템플릿

```markdown
# Issue: {task_name}

- 발견일: {YYYY-MM-DD}
- 발견 위치: {테스트 결과 파일 참조}
- 심각도: critical / high / medium / low
- 관련 파일: {영향 받는 파일 목록}

## 문제 설명

{무엇이 문제인지}

## 재현 방법

{어떻게 하면 재현되는지}

## 예상 원인

{추정 원인}
```

### 해결 시 — 파일명에 `_solved` 접미어 추가 + 해결 내용 덧붙임

```markdown
# Issue: {task_name} [SOLVED]

- 발견일: {YYYY-MM-DD}
- 해결일: {YYYY-MM-DD}
- 발견 위치: {테스트 결과 파일 참조}
- 심각도: critical / high / medium / low
- 관련 파일: {영향 받는 파일 목록}

## 문제 설명

{원본 유지}

## 재현 방법

{원본 유지}

## 예상 원인

{원본 유지}

---

## 해결 내용

- 해결 방법: {무엇을 어떻게 수정했는지}
- 변경 파일: {수정된 파일 목록}
- 검증: {해결 확인 방법}
```

---

## 규칙

- 테스트 없이 작업 완료 선언 금지
- 테스트 결과 파일은 삭제 금지 — 누적 기록
- 이슈는 해결될 때까지 `issues/` 폴더에 유지
- 해결 시 파일 이동 아님 — **파일명 변경**(`_solved` 접미어)
- 이슈 파일에 원본 내용 삭제 금지 — 해결 내용을 하단에 추가
